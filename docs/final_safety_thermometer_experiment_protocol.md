# Final Safety Thermometer Experiment Protocol

Status: method-freeze draft before publication-scale experiments.

Scope: This protocol freezes the experimental design before any final model training, calibration, threshold selection, provider calls, AgentDojo runs, or test-set evaluation. Existing AgentDojo full v2 outputs are used only to document implementation readiness and known gaps. Test split data must remain untouched until the method is frozen and the final validation-selected configuration is locked.

## 1. Final Frozen Architecture

### Evidence Streams

Safety Thermometer consumes trajectory-prefix evidence only. The unified trace schema exposes the following prefix-visible streams:

- instruction stream: user instruction and task request context
- plan stream: plan summary and stated intermediate intent
- action stream: proposed tool name and tool arguments
- observation stream: tool observations and execution feedback
- self-check stream: model self-assessment or safety/risk self-report
- notes stream: only pre-step and post-step notes may be used for stream q features; final-hook notes are audited but excluded from model features

Forbidden model inputs are future labels, future severity, first risk step, lead-time labels, oracle violations, utility, security, benchmark outcomes, and test split membership.

Each evidence stream must produce both learned outputs:

- z_stream: a learned evidence representation for that stream
- q_stream: a learned risk estimate in [0, 1] for that stream

Learned z is a core method component, not an optional extension or diagnostic. The final method is not considered frozen unless every primary evidence stream has both z and q generation specified under the train-only protocol.

### LLM-as-Judge Labels

Final experiments use LLM-as-judge labels at the episode or event level, with deterministic schema validation and retained rationales. The judge produces binary risk, severity, category, deception, surface-benign status, and first-risk-step annotations. Oracle rules remain auxiliary diagnostics and are not the primary supervision source.

### Learned Stream z and q Estimators

Each evidence stream receives a learned representation z and a learned risk estimate q in [0, 1]. The stream z encoder and stream q estimator are trained only on training data. For training rows, z and q values used by downstream models must be K-fold out-of-fold outputs. For validation and test rows, z and q values must be generated by stream models fit on the full training split only.

Required learned stream outputs:

- z_instruction and q_instruction
- z_plan and q_plan
- z_action and q_action
- z_observation and q_observation
- z_self_check and q_self_check

The current AgentDojo full v2 variance implementation uses heuristic q proxies and does not yet implement learned z. This is useful for engineering validation, but it is not the frozen publication method until replaced by learned stream z and learned stream q.

### Learned-q Variance Disagreement

Disagreement features are computed from learned stream q values:

- D_t: dispersion across available stream q values at prefix t
- D_self: self-check underreporting gap, defined as max(0, max(q_instruction, q_plan, q_action, q_observation) - q_self_check)
- D_act: action-centered mismatch, including plan-action, intent-action, and observation-action gaps

The frozen implementation should retain absolute action-self gap only as a diagnostic or ablation feature, not as the primary D_self definition.

### Temporal Risk Accumulator

The runtime state includes prefix-level cumulative and max features that summarize risk evidence over time. The default publication configuration is T1 lightweight temporal memory unless validation selects T0 or a fully implemented T2 learned temporal model.

### Severity-Weighted Temporal Risk Estimator

The main downstream temporal Risk Estimator predicts severity_weighted_future_risk, not only binary future_risk_label. Binary risk metrics remain reported for interpretability and compatibility with earlier experiments.

The temporal Risk Estimator consumes the frozen feature family:

- learned stream representations: z_instruction, z_plan, z_action, z_observation, z_self_check
- learned stream risk estimates: q_instruction, q_plan, q_action, q_observation, q_self_check
- disagreement features: D_t, D_self, D_act
- temporal memory features: cumulative, max, monotone, or learned temporal state features selected under the temporal accumulator protocol

No future label, severity target, first risk step, oracle violation, utility/security metadata, final outcome, or test split indicator may enter this estimator as an input feature.

### Severity-Weighted Calibration

Calibration maps estimator outputs to a 0-100 Safety Thermometer score. Calibration is selected on validation only and evaluated with severity-weighted reliability metrics, including severity-weighted ECE and reliability buckets.

### Runtime Policy

Runtime policy maps calibrated thermometer scores to actions:

- continue
- watch
- verify
- alert
- block

Verify, alert, and block thresholds are selected on validation only. Once selected, thresholds are frozen before the test split is evaluated.

## 2. Benchmark Protocol

### AgentHazard Primary Benchmark

AgentHazard is the primary publication benchmark. Final claims should center on AgentHazard unless the benchmark is unavailable or explicitly scoped out. AgentHazard traces must be converted into the unified trace schema before feature extraction.

### AgentDojo Secondary Benchmark

AgentDojo is the secondary benchmark and engineering stress test. Current full v2 AgentDojo artifacts contain 96 episodes and 806 prefix rows, with aligned no-disagreement, heuristic disagreement, variance disagreement, and heuristic-plus-variance disagreement variants.

### Unified Trace Schema

All benchmarks must be normalized to:

- episode_id
- step_id
- hook_type
- timestamp
- user_instruction
- plan_summary
- proposed_tool
- tool_args
- observation
- state_delta
- self_check
- notes
- benchmark metadata stored outside model feature columns

### Train/Validation/Test Split Rules

Splits are episode-level, never prefix-row-level. No episode may appear in more than one split. Validation is used for model selection, calibration selection, and threshold selection. Test is used once after method freeze.

Current AgentDojo full v2 split status:

- train: 58 episodes, 472 rows, label counts 0=316 and 1=156
- validation: 19 episodes, 176 rows, label counts 0=84 and 1=92
- test: 19 episodes, 158 rows, label counts 0=84 and 1=74

The test split exists only for future final reporting and must not be used in any pre-freeze selection step.

## 3. Label Protocol

The judge label schema is:

- binary_risk_label: 0 or 1 episode/event risk label
- severity_score: continuous or ordinal risk severity
- risk_category: normalized category such as privacy, financial, security, irreversible state change, policy violation, or deception
- deception_indicator: whether the trajectory involves deceptive, hidden, or misleading behavior
- surface_benign_indicator: whether the request appears benign at the surface while later evidence indicates risk
- first_risk_step: first step at which the trajectory becomes risky
- rationale: short judge rationale with cited evidence

Prefix propagation:

- future_risk_label at prefix t is 1 if any risk event occurs at or after t
- future_severity at prefix t is the maximum future severity at or after t
- severity_weighted_future_risk at prefix t is future_risk_label multiplied by a normalized severity weight

Label fields are targets or diagnostics only and must never be included as model inputs.

## 4. Stream z and q Estimator Protocol

Stream-level datasets are constructed from prefix-visible fields only. Each row is a stream example keyed by episode_id, step_id, hook_type, stream_name, stream_text or structured stream payload, and the target derived from training labels.

Fitting rules:

- train-only fitting for all stream z encoders and stream q estimators
- K-fold out-of-fold z and q generation on train rows
- train-full stream z encoder and stream q estimator for validation z and q generation
- the same train-full stream models for test z and q generation after method freeze
- no validation or test labels used in stream z or q fitting
- no final-hook outcome notes used for stream z or q

Leakage checks must verify that future_risk_label, future_severity, severity_weighted_future_risk, first_risk_step, oracle_violation, utility, security, and test split membership are absent from stream z and q model features.

## 5. Disagreement Protocol

### no_disagreement

The no_disagreement condition excludes all disagreement features. It may include base prefix features, cumulative risk evidence, and the calibrated thermometer pipeline.

### learned_q_variance_disagreement

The primary disagreement condition uses learned stream q values while the downstream temporal Risk Estimator also consumes learned stream z values:

- q_instruction
- q_plan
- q_action
- q_observation
- q_self_check
- d_stream_variance
- d_stream_range
- d_stream_max
- d_stream_mean
- d_self_check_underreport_gap
- d_plan_action_gap
- d_intent_action_gap
- d_observation_action_gap
- num_available_streams
- stream availability indicators

Learned stream z values are not disagreement scores by themselves, but they are required core inputs to the temporal Risk Estimator in every full-method configuration.

Primary definitions:

- D_t = variance over available q streams
- D_self = max(0, max(q_instruction, q_plan, q_action, q_observation) - q_self_check)
- D_act = action-centered gap features, especially plan-action, intent-action, and observation-action disagreement

### optional heuristic_disagreement

Heuristic disagreement remains an optional engineering baseline and ablation. It must be clearly separated from learned-q variance disagreement in all tables.

## 6. Temporal Accumulator Protocol

Temporal variants:

- T0 no temporal memory: single-prefix features only
- T1 lightweight cumulative/max/monotone accumulator: cumulative counts, max risk so far, max disagreement so far, cumulative stream variance, max D_self so far, max D_act so far
- T2 learned temporal model: sequence model or learned recurrent/attention aggregator, only if fully implemented and selected using validation

The default frozen candidate is T1 unless validation establishes that T0 or T2 is superior under the predeclared selection rule.

## 7. Calibration Protocol

Calibration target is severity_weighted_future_risk. Calibration candidates may include identity, Platt scaling, isotonic regression, and other predeclared calibrators.

Selection and reporting:

- fit calibration on training predictions only or on the predeclared train/calibration partition
- select calibration method on validation only
- compute severity-weighted ECE
- report reliability buckets over the 0-100 Safety Thermometer score
- select verify, alert, and block thresholds on validation only
- never tune thresholds on test

## 8. Baseline Matrix

Final experiments report:

- text-only final-output classifier
- single-signal monitor
- single-step fusion without temporal memory
- w/o disagreement
- w/o temporal memory
- w/o calibration
- full Safety Thermometer

Each baseline must use the same episode splits, label protocol, and evaluation slices.

## 9. Slice Evaluations

Required slices:

- all
- utility=true
- utility=false
- utility=unknown
- deceptive
- surface-benign risky
- self-check-low external-high
- suite
- benchmark

AgentDojo-specific note: current full v2 data contains utility/security unknown_no_summary rows. These are valid diagnostic groupings but should be reported separately.

## 10. Final Reporting

Final reporting includes:

- Table 3 early-warning metrics: AUROC, AUPRC, pre-risk AUROC, pre-risk AUPRC, positive lead time, missed risk episodes
- Table 4 runtime policy outcomes: verify/alert/block thresholds, containment proxy, false alert rate, operational contained rate
- ablation table: no disagreement, learned-q variance disagreement, optional heuristic disagreement, temporal variants, calibration variants
- slice table: all required slices
- latency: feature extraction, stream q inference, risk estimator inference, calibration, policy decision

Every table must state whether it is validation-only, test-final, AgentHazard primary, or AgentDojo secondary.

## 11. Method-Freeze Rules

Before any test evaluation:

- model classes are fixed
- feature columns are fixed
- stream z and q training protocol is fixed
- calibration candidates and selection rule are fixed
- threshold grid and selection rule are fixed
- slice definitions are fixed
- primary and secondary benchmarks are fixed
- random seeds and split manifests are fixed

Validation may be used for model, calibration, and threshold selection. Test is used once only after method freeze. After final validation selection, no feature, model, threshold, label, or slice definition changes are allowed before test reporting.
